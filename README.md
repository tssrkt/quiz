# Quiz / Викторины

Статический русскоязычный сайт викторин на HTML, CSS, JavaScript и JSON. Контент редактируется через Pages CMS, проверяется Python-скриптом без сторонних зависимостей и автоматически публикуется в GitHub Pages через GitHub Actions.

## Как устроена публикация

Каждый тег хранится в `data/tags/<slug>.json`, каждая викторина — в `data/quizzes/<slug>.json`. Ручного списка файлов нет.

При изменении ветки `main` workflow `.github/workflows/pages.yml`:

1. запускает `python tools/build_site.py`;
2. сканирует все JSON в обоих каталогах;
3. проверяет теги, викторины, ответы, ссылки и изображения;
4. создаёт `_site/data/catalog.json` только с опубликованными викторинами;
5. копирует в `_site` только файлы готового сайта;
6. загружает `_site` как GitHub Pages artifact;
7. публикует artifact официальным действием GitHub Pages.

Если проверка не проходит, artifact не создаётся и повреждённая версия не заменяет предыдущую публикацию. Сгенерированная папка `_site` не коммитится.

## Структура

```text
index.html                     главная
quizzes.html                   каталог
quiz.html                      прохождение и результат
contacts.html                  контакты
css/style.css                  стили
js/common.js                   меню и общие функции
js/quizzes.js                  теги, сортировка, URL и пагинация
js/quiz.js                     прохождение, прогресс и результат
data/tags/*.json               теги
data/quizzes/*.json            полные викторины
img/covers/                    обложки
img/quiz/                      изображения вопросов
tools/build_site.py            проверка и сборка
examples/quiz-template.json    пример, не попадающий в каталог
.pages.yml                     форма Pages CMS
.github/workflows/pages.yml    публикация GitHub Pages
```

## Подключение Pages CMS

1. Откройте [app.pagescms.org](https://app.pagescms.org/) и войдите через GitHub.
2. Установите GitHub App Pages CMS и разрешите доступ к `tssrkt/quiz`.
3. Выберите репозиторий и ветку `main`.
4. Pages CMS прочитает `.pages.yml` и покажет коллекции «Теги» и «Викторины».

CMS сохраняет изменения непосредственно в GitHub, создавая коммиты. После изменения `main` GitHub Actions автоматически проверит и соберёт сайт. Новая корректная опубликованная викторина появится в каталоге без редактирования JavaScript, HTML или списка файлов.

## Создание тега

1. В Pages CMS откройте «Теги» и нажмите создание записи.
2. Заполните название и уникальный `slug` строчными латинскими буквами, цифрами и дефисами.
3. Укажите целый неотрицательный порядок отображения.
4. Включите «Опубликован», если тег должен появиться в каталоге.
5. Сохраните запись. Имя файла автоматически формируется как `<slug>.json`.

## Создание викторины

1. Создайте запись в коллекции «Викторины».
2. Заполните название, уникальный `slug`, дату, краткое описание и вступление.
3. Оставьте «Опубликована» выключенным, пока это черновик.
4. Выберите один или несколько тегов из коллекции тегов.
5. При необходимости выберите или загрузите квадратную обложку. Она сохраняется в `img/covers`.
6. Добавьте вопросы. Их можно удалять и переставлять в повторяющемся списке.
7. Для изображения вопроса выберите существующий файл либо загрузите новый в `img/quiz`. Можно выбирать файлы из вложенных каталогов.
8. Добавьте от 2 до 6 вариантов. Варианты можно удалять и переставлять.
9. Для каждого варианта заполните стабильный уникальный `id` и текст.
10. Включите флаг «Правильный ответ» ровно у одного варианта вопроса.
11. Заполните объяснение и при необходимости сведения об источнике.
12. Сохраните викторину и проверьте запуск workflow во вкладке **Actions** репозитория.

Pages CMS не создаёт надёжно отдельную папку изображения из slug викторины. Поэтому используется общий `img/quiz` с выбором файлов из существующих вложенных папок.

## Формат ответа

Правильность хранится внутри варианта, поэтому изменение порядка не повреждает правильный ответ:

```json
{
  "id": "question-01",
  "question": "Текст вопроса",
  "image": "img/quiz/example.webp",
  "image_alt": "Описание изображения",
  "answers": [
    {"id": "answer-01", "text": "Первый вариант", "correct": false},
    {"id": "answer-02", "text": "Правильный вариант", "correct": true}
  ],
  "explanation": "Почему этот ответ правильный.",
  "image_source": "",
  "image_author": "",
  "image_source_url": ""
}
```

Используется только этот формат; отдельного `correct_answer` нет. Полный образец находится в `examples/quiz-template.json` и не сканируется сборщиком.

## Черновики и предварительный просмотр

Черновик с `published: false` не включается в каталог и без специального параметра не запускается. После успешной сборки его можно открыть напрямую:

```text
https://tssrkt.github.io/quiz/quiz.html?quiz=my-quiz&preview=1
```

Локально адрес выглядит так:

```text
http://localhost:8000/quiz.html?quiz=my-quiz&preview=1
```

На странице показывается предупреждение «Предварительный просмотр. Викторина не опубликована». Для публикации включите поле «Опубликована» и сохраните запись.

`preview=1` — не защита приватных материалов. Полный JSON черновика находится среди статических файлов и доступен технически подготовленному посетителю.

## Прохождение и сохранение прогресса

Движок сохраняет попытку отдельно для каждого slug в `localStorage`. Сохранение содержит версию формата, сигнатуру вопросов и вариантов, текущую позицию, выбранные ответы, их правильность, количество правильных ответов, дату и признак завершения. Если вопросы или варианты изменились, несовместимое сохранение автоматически сбрасывается.

Правильный ответ блокирует варианты и автоматически переводит дальше через 800 мс. Неправильный ответ показывает правильный вариант, объяснение и кнопку перехода. Анимация конфетти встроена локально и отключается при `prefers-reduced-motion`.

На итоговом экране доступны Web Share, Telegram, ВКонтакте и копирование результата. Публикуемая ссылка ведёт непосредственно на викторину и не содержит `preview` или данных прогресса.

## Локальная проверка и запуск

Только проверить контент:

```bat
python tools/build_site.py --check
```

Собрать и запустить:

```bat
python tools/build_site.py
python -m http.server 8000 -d _site
```

Откройте `http://localhost:8000/`. Прямое открытие через `file://` не поддерживается, поскольку браузеры ненадёжно разрешают загрузку JSON в таком режиме.

Автоматические тесты:

```bat
python -m unittest tests.test_build_site
node tests/catalog.test.js
node tests/quiz.test.js
```

При наличии Python Playwright и Chrome три полных браузерных сценария запускаются командой:

```bat
python tests/ui_quiz_scenarios.py
```

## Что проверяет сборщик

- обязательные поля и формат slug;
- уникальность slug викторин и тегов;
- уникальность названий тегов без учёта регистра;
- совпадение slug с именем JSON-файла;
- дату публикации;
- существование используемых тегов;
- вопросы и уникальность их `id`;
- от 2 до 6 вариантов и уникальность их `id` внутри вопроса;
- ровно один правильный вариант;
- обязательное объяснение;
- относительные пути обложек и изображений;
- существование локальных изображений;
- внешние ссылки источников с `http://` или `https://`.

Ошибки выводятся с именем файла и путём проблемного поля. Сборка завершается ненулевым кодом.

## Настройка GitHub Pages

В **Settings → Pages → Build and deployment → Source** выберите **GitHub Actions**. Workflow использует минимальные разрешения: чтение содержимого для сборки и `pages: write`/`id-token: write` только для задания публикации. Сгенерированные файлы обратно в `main` не коммитятся.

После сохранения контента проверяйте **Actions → Build and deploy GitHub Pages**. Зелёный запуск означает, что проверка, сборка artifact и публикация завершились успешно.

## Ограничения

- Pages CMS не может гарантировать условие «ровно один правильный ответ» внутри формы; это строго проверяет сборщик до публикации.
- Предпросмотр черновика публичен по своей природе.
- Прогресс хранится только в `localStorage` текущего браузера и устройства.
